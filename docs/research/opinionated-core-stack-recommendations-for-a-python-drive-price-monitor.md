# Opinionated Core Stack Recommendations for a Python Drive-Price Monitor

## Bottom line

For this specific app, I would choose **Django for the web framework**, **PostgreSQL for the database**, and **uv for environment and dependency management**. The core reason is not raw request throughput; it is **total build effort for a single maintainer** building a CRUD-heavy, data-heavy, authenticated web app with internal back-office needs, background ingestion, alerts, and a search/filter dashboard. Django’s built-ins map directly onto those needs, PostgreSQL is the better fit for mixed relational-plus-history-plus-search data, and uv is the strongest 2026 choice for fast, reproducible Python application workflows.

If I reduce the three decisions to the tradeoff that matters most, it looks like this: **Django trades some API-first elegance for a large reduction in application assembly work; PostgreSQL trades a little simplicity for meaningfully better long-term fit for JSON, search, and time-series-ish history; uv trades some version-churn risk for the best speed and developer ergonomics in the current Python tooling landscape.**

## Web framework recommendation

My recommendation is **Django, with server-rendered templates and HTMX**, not FastAPI as the primary web framework. Your preference for FastAPI is still correct in one important sense: it remains the cleaner choice when the API is the product, when OpenAPI and generated clients are first-class, or when the service’s main value is typed request/response contracts exposed to other systems. But for *this* app, the center of gravity is not “API platform”; it is “authenticated listings database plus dashboards plus CRUD plus alerts plus internal operations.” Django’s built-in admin, auth, models/ORM, and migrations materially reduce the amount of software you have to assemble before you even get to your domain logic.

Django’s advantages line up unusually well with your feature set. The admin is explicitly designed as a quick, model-centric internal management interface for trusted users, which is exactly what you want for inspecting scraped offers, correcting bad matches, triaging ingestion issues, reviewing users and alerts, and managing catalog data; Django’s own docs also warn that the admin is *not* meant to be your entire front end, which is the right separation for this app. Its auth system is built to cover common project needs with passwords and permissions, and its migrations are “mostly automatic,” which matters a lot in a schema that will keep evolving as you refine scraping sources, normalization, scoring, and alert logic.

FastAPI’s strengths are real, but they do not erase the assembly tax. FastAPI is built around APIs, automatic OpenAPI docs, interactive documentation, and typed contracts that can drive generated SDKs. It also lets you render Jinja templates and use any database or ORM you want, which is a strength for custom stacks. The problem is that, for this application, “you can use anything” mostly means **you must choose and integrate everything**: ORM, migrations, auth/session model, admin/back-office tooling, form handling patterns, and probably frontend conventions. That freedom is worth it when the app is API-first; it is less attractive when most of your work is operational CRUD, dashboarding, and internal tooling.

FastAPI’s async model is also less decisive here than it first appears. Django now has an async-enabled request stack under ASGI, and its docs say many parts of Django provide async APIs, including the ORM, cache, authentication, sessions, and signals. That does not make Django “the same as FastAPI,” but it narrows the gap enough that framework-level async is no longer a decisive argument for a listings dashboard app. More importantly, your heavy lifting is the ingestion pipeline and alert/background processing, not the interactive request path; FastAPI’s own docs say that for heavy background computation you may want a separate tool like Celery rather than in-process background tasks. That means the architectural answer for scraping and alerts is “use a worker,” regardless of whether the web layer is Django or FastAPI.

Flask is the weakest fit of the three. Flask is intentionally lightweight, WSGI-based, template-friendly, and flexible. Its maintainers explicitly describe it as offering suggestions rather than enforcing dependencies or project layout, with many features added by community extensions. That is a great profile when you want a minimal core and are happy curating your own stack. For this project, though, that mostly means more decisions, more glue code, and more long-term maintenance surface for one maintainer—not more leverage. Flask is viable; it is just hard to justify over Django for this shape of application, and hard to justify over FastAPI if you actually want API-first ergonomics.

For the UI layer, I would not build a SPA first. Django templates already align well with Django’s own contrib apps, and HTMX’s model is deliberately HTML-oriented and close to normal, non-AJAX web apps. For filter forms, paginated listings, search results, inline updates, lightweight modal/detail panes, and alert configuration screens, **server-rendered HTML plus HTMX** is the highest-leverage choice: less JavaScript state, fewer moving parts, simpler auth/CSRF/form workflows, and still enough interactivity for a modern dashboard. If you chose FastAPI anyway, I would still lean toward Jinja2 plus HTMX before reaching for a separate SPA, unless the UI becomes substantially richer than the feature set you described.

So the direct answer to your existing preference is this: **your instinct toward FastAPI is right in general, but I would override it for this app and pick Django**. What you give up is FastAPI’s superior API-first ergonomics—automatic docs, generated-client friendliness, and a more obviously API-native programming model. What you gain is a large reduction in scaffolding work: built-in auth, permissions, ORM, migrations, admin, and a template stack that works naturally with HTMX. For a single-maintainer listings/alerts product, that is a good trade.

## Database recommendation

My recommendation is **PostgreSQL**, and it is not especially close for this workload. You are storing normalized entities such as products, merchants, offers, users, and alerts; append-heavy price observations over time; probably some raw source payloads; and likely some search text or fuzzy matching fields. PostgreSQL is a better default for that mixed workload because its JSONB story is stronger, its full-text and extension ecosystem is richer, and its time-oriented operational options are more compelling when you eventually outgrow a single plain table.

The JSON difference is meaningful here. PostgreSQL exposes both `json` and `jsonb`, and its docs are explicit that `jsonb` stores data in a decomposed binary format that is significantly faster to process than reparsing text each time; they also explicitly note that `jsonb` supports indexing. That is a very natural fit for storing raw scrape/API payloads, normalized-enough metadata, and occasionally querying into payload structure without turning the raw table into your primary domain model. MySQL also has a native JSON type with validation and an efficient internal format, so it is not weak here—but PostgreSQL’s JSONB plus indexing story is simply the stronger and more widely-used fit for this style of application.

Search is another place where PostgreSQL fits the app better. PostgreSQL’s built-in full-text search is designed to identify natural-language documents that satisfy a query and sort them by relevance, and the `pg_trgm` extension can add fast similarity search for “close” text matches. That combination is useful for product-title normalization, merchant-name fuzziness, model-capacity/brand matching cleanup, and user-facing search that needs to be a little forgiving without forcing an external search engine. MySQL does have native full-text search via `MATCH() ... AGAINST()`, so this is not a capability gap, but PostgreSQL gives you a more flexible search toolbox inside the database.

Partitioning and time-series-ish history also favor PostgreSQL. PostgreSQL’s current manual documents built-in declarative partitioning, where a partitioned table routes rows to ordinary child tables based on the partition key. That is exactly the mechanism you would eventually use for a large `price_history` or `offer_price_event` table partitioned by month or quarter. MySQL also supports partitioning and gets the usual maintenance benefits, including easier dropping of old data and partition pruning for some queries. The critical problem is that Oracle’s own MySQL docs explicitly say full-text searches are **not supported for partitioned tables**. For an app that plausibly wants *both* time-based partitioning *and* text search over listings/products, that restriction is a real design wart, not a minor footnote.

At hobby or small-business scale, though, I would **not** start with TimescaleDB, and I might not start with partitioning either. This is partly judgment, but it is judgment grounded in what the docs imply about the tooling. PostgreSQL already gives you plain relational tables, JSONB, full-text search, and native partitioning if you need it. BRIN indexes are explicitly described as being designed for **very large tables** where a column is naturally correlated with physical storage, which is often true for append-oriented timestamp data. That is a hint to start simple: a normal history table, sensible B-tree indexes for your common lookups, and perhaps BRIN later if the time table becomes large enough to justify it. You can add partitions when retention or maintenance becomes painful; you do not need to pay that complexity tax on day one.

TimescaleDB becomes justified when you know you need its opinionated time-series automation. Its docs highlight hypertables, background-refreshed continuous aggregates, compression and columnstore-style storage, and retention/compression policies. Those are excellent features once ingest volume, retention length, and reporting latencies become operational problems. But that is exactly why I would treat it as a **second-stage optimization** for your app, not the default. For a single-maintainer hard-drive monitor, native PostgreSQL almost certainly carries you a long way before hypertables, continuous aggregates, and lifecycle policies earn their operational weight.

So the practical database recommendation is: **PostgreSQL 18, ordinary relational schema, JSONB for raw payload storage, no TimescaleDB initially, and no partitioning until the history table actually needs it**. What you give up versus MySQL is mostly familiarity if you already prefer the MySQL ecosystem. What you gain is the cleaner long-term path for rich querying, fuzzy search, full-text, JSONB indexing, and future partitioning without the MySQL full-text/partitioning conflict.

## Environment and dependency management recommendation

My recommendation is **uv** for a 2026 Python application project. Astral’s own docs now position uv as a single tool that can replace or cover major parts of `pip`, `pip-tools`, `pipx`, `poetry`, `pyenv`, `twine`, `virtualenv`, and more; they also document a universal lockfile, Python version management, project management, export capabilities, and direct CI integration. If you are building and deploying an application, not publishing a reusable library as your primary goal, that is a very strong package.

The biggest practical uv advantages for your setup are reproducibility and workflow compression. uv’s lockfile is documented as using a **universal resolution** portable across operating systems, architectures, and Python versions, and the lock/sync workflow is first-class rather than bolted on. That matters even on a single Debian VM because you are still likely to have multiple dev environments over time, and because reproducible deploys are easier when “developer machine,” “CI runner,” and “server” all consume the same lock state. uv can also export lockfiles to other formats if you need interoperability with existing workflows.

uv also has the best current CI story of the three options you asked about. The docs include a dedicated GitHub Actions integration guide, recommend the official `setup-uv` action, and cover installation, Python setup, syncing/running, and caching. That means the “use it locally, use the exact same thing in CI, pin it, and run frozen/synced installs” workflow is now normal rather than aspirational. For a 2026 app project, that matters more than abstract packaging philosophy.

Poetry is still a solid option, but I would rank it second. Poetry’s current docs still present it as a dependency-management-and-packaging tool with a lockfile for repeatable installs, and its install/sync workflow is mature and understandable. It is especially attractive when packaging and publishing are central because its docs directly cover building for distribution and working with PyPI and private repositories. If your app were also intended to become a reusable package-first product, or if your team were already standardized on Poetry, I would not call Poetry a mistake. I just think uv now offers the better operational center of gravity for application projects.

`pip-tools` plus `venv` is the conservative, low-magic option, and it remains respectable. The stdlib’s `venv` creates isolated environments; `pip-compile` can turn dependencies from `pyproject.toml` or other inputs into locked requirements; and `pip-sync` will make an environment match those requirements exactly. If you want the narrowest tool surface, the least opinionated workflow, and easy mental mapping to standard Python packaging primitives, this route is still very defensible. The downside is that it is a parts bin, not a unified workflow: environment creation, Python version management, lock generation, syncing, and package publishing all remain separate concerns.

So my recommendation here is straightforward: **use uv unless you have a specific reason to prefer Poetry’s packaging workflow or pip-tools’ minimalism**. The key tradeoff is that uv is the more fast-moving project, so commands and recommended patterns are somewhat more version-sensitive than the older tools. In exchange, you get the best combination of speed, lockfile ergonomics, Python management, exportability, and CI integration currently available in the official docs.

## Recommended stack summary

**Recommended stack:** Django as the primary web framework; Django templates for server-rendered pages; HTMX for incremental interactivity instead of a SPA; PostgreSQL 18 as the database; JSONB columns for raw ingestion payloads; ordinary relational tables for products, merchants, offers, users, and alerts; a simple append-oriented price-history table that starts unpartitioned and moves to native PostgreSQL range partitioning only if growth demands it; uv with a committed `uv.lock` for local, CI, and deployment reproducibility; and a separate worker process for ingestion and alert jobs rather than relying on in-request background helpers. That stack is the best balance of build speed, operational simplicity, and future headroom for the app you described.

## Version-sensitive notes

A few parts of this recommendation are explicitly version-sensitive as of **July 3, 2026**. Django’s current stable line is 6.0, released on December 3, 2025, and Django 5.2 added important async-auth work that helps narrow the historical FastAPI-vs-Django async gap. Flask’s stable docs are 3.1.x, and the project’s GitHub page shows 3.1.3 as the latest release in February 2026. Those specifics matter because framework advice around async support and built-ins can age quickly.

For databases, PostgreSQL’s current official manual is version 18, while MySQL’s download pages show both **9.7.1 LTS** and **8.4.10 LTS** available. The concrete MySQL feature references I used for JSON, partitioning, and full-text behavior come largely from the 8.4 and 8.0 reference pages because Oracle documents those behaviors explicitly there, including the important restriction that full-text search is not supported on partitioned tables. If Oracle changes that restriction in later 9.x documentation, that would narrow—but not eliminate—the practical gap for this app.

The most version-sensitive area is Python tooling. uv’s docs have been changing rapidly through 2025 and 2026, which is normal for a fast-moving tool still expanding scope; Poetry is currently at 2.4.1 in its official history; and pip-tools’ current docs reflect an actively maintained 7.5.x line. So the recommendation for uv is strong, but it is also the one I would treat as most worth re-validating against current docs at implementation time if you are standardizing commands for CI, release automation, or team onboarding.
