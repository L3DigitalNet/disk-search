# SSD Vendor Part-Number (MPN) Decoding Schemes and Prior-Art Model-to-Spec Bootstrap Databases

## Bottom line

SSD MPN decoding is **more fragmented across vendors than the HDD side** (see the companion HDD report, [`programmatic-identity-and-warranty-verification-for-used-enterprise-hdd-listings.md`](programmatic-identity-and-warranty-verification-for-used-enterprise-hdd-listings.md), for the Seagate/Toshiba/WD HDD decoders). Only **two vendors publish a true official generic decoder page**: **Kingston** (a dedicated, current, per-series "SSD Part Number Decoder" web page) and, historically, **Micron** (an official PDF numbering-system guide, but it only documents generations through roughly the M500/M600/510DC/5100 era — there is no current public Micron guide covering the 7450/9550/9650-class parts). **Samsung, SK hynix, Seagate (Nytro), and Kioxia have no official generic decoder**; the only reliable per-model source for these is the vendor's own datasheet/product-brief for that specific family, cross-checked against community teardown/forum/reseller listings. **Solidigm** sits in between: it publishes an official mapping from legacy Intel marketing names to current Solidigm marketing names, and its Product Change Notification (PCN) documents show the actual orderable part number (still `SSDPxxxx`-style, Intel-legacy grammar) alongside the new `D3/D5/D7` marketing name — so the *marketing* name changed but the *underlying MPN grammar* did not.

For the bootstrap-dataset question: **smartmontools' `drivedb.h`** is GPL-2.0-or-later and is explicitly *not* a spec catalog — it is a `modelregexp -> family name + SMART-attribute-threshold presets` lookup, nothing about capacity/interface/form-factor. It is safe to use **only via runtime invocation of a separately-installed system `smartctl` binary** (subprocess boundary, no vendoring), never by copying the file's regex/preset table into hw-radar's own MIT-licensed repository, which would put GPL-2.0 content inside a repo whose CI license gate allowlists only MIT/BSD/Apache/LGPL-3.0. **Open Icecat** has real brand/GTIN-keyed spec coverage and a genuinely free tier, but its "Open Content License" is a client-agreement structure aimed at *channel partners*, not clearly a right to freely redistribute or embed scraped/derived content in a public source repository — treat it as an enrichment lookup called at runtime against Icecat's hosted service, not as data to vendor. **TechPowerUp's SSD database** is high-quality and component-level (NAND type, controller, package) but has **no self-serve API for this dataset** (the documented free API is GPU/CPU-only); commercial/non-academic access requires directly emailing the maintainer with no published pricing — not viable as an unattended bootstrap source for a personal tool without that conversation happening first.

Micron's marketing name (e.g. "9550 PRO"/"9550 MAX") is not derivable from the part number without that datasheet. | **Official generic decoder exists but is generation-stale.** Current-gen family/spec mapping is **official-but-per-family only** (no current generic legend). |
| **Solidigm** | Official Intel support article *"How to Identify a Formerly Intel NAND SSD (now Solidigm SSD) Product"* gives the full legacy-name → Solidigm-name mapping table (e.g. `Intel SSD D7-P5520 Series` = `Solidigm D7-P5520 Series`). Official Product Change Notification (PCN) PDFs (e.g. `PCN0000017185-02.pdf`) show the **actual orderable part number** alongside the marketing name, e.g. `D7-P5520 Series (3.84TB, E1.S…)` = part number `SSDPFVKX038T11Z`. Family pages (`solidigm.com/products/data-center/d7/p5520.html`) list "Order Codes." | The Intel-legacy `SSDPxxxxxxxxxxxx` grammar (`SSD` + 2-letter series + form-factor/generation letters + capacity token + suffix) still reliably decodes brand + rough capacity + a form-factor/series hint, and this grammar **did not change** when Intel's NAND business became Solidigm — only the marketing name (`D3-`/`D5-`/`D7-` + model number) changed. The marketing name's `D3`/`D5`/`D7` prefix reliably signals SATA-legacy / mid-endurance-PCIe / high-performance-PCIe tier respectively (inferred consistently across the D3-S4520, D5-P5316, D7-P5520/P5620/P5810/PS1010 family pages). | No single public page gives the **character-by-character grammar** of the current `SSDPxxxx` orderable code the way Micron's or Kingston's decoders do — it must be inferred from PCN tables and datasheets per family. The `D3/D5/D7` marketing-name tier mapping is **corroborated across multiple official family pages but not itself published as an explicit legend** — treat the tier inference as official-data-backed but community-synthesized. | **Legacy→current name mapping: official.** Orderable-code grammar: **official data, no consolidated legend** (must cross-reference PCNs/datasheets per family). |
| **SK hynix** | No official generic decoder found. Marketing-name family pages (skhms.com company history; storagereview.com PE6011/PE8010 reviews) give the `PE####`/`PC###`/`BC###` marketing-series names. The **actual physical part numbers use a separate `HFS`/`HFM`-prefixed grammar** (e.g. `HFS3T8GECTX098N`, `HFM512GD3HX015N`), documented incidentally in official NIST FIPS 140-2/140-3 Security Policy PDFs for specific SED models (e.g. PE8010/PE8030) and cross-checked by third-party data-recovery shop reference material (Rossmann Group) and reseller listings (eBay). | `HFS`-prefix parts are SATA-or-NVMe SK hynix drives; `HFM`-prefix parts are specifically NVMe (per the Rossmann Group reference, `[blog]`-grade but corroborated by matching part numbers on multiple independent reseller/eBay listings for the same named products, e.g. PC711/HFS512GDE9X080N, PC801/HFS001TEJ9X101N). A capacity token follows the prefix (`256`, `512`, `001T`, `002T`, …). | There is **no official public legend** mapping the remainder of the `HFS…`/`HFM…` string (controller/NAND-generation/suffix letters) to spec attributes; every field beyond brand+prefix-class+rough-capacity requires the specific datasheet or FIPS security-policy document for that exact model. Marketing name (PE8010, PC711, …) is not derivable from the physical part number without an external cross-reference table. | **No official generic decoder.** Prefix-class (HFS vs HFM) is **community-decoded but corroborated across 2+ independent sources** (repair-shop reference + multiple matching reseller listings + official FIPS docs showing the same part-number shape). |
| **Kioxia** | No official generic decoder page, but official enterprise/data-center datasheets (`EnterpriseSSD_DataSheet_E.pdf`, `DataCenterSSD_DataSheet_E.pdf`) list full "Model Number" columns per capacity/interface/form-factor, and official per-series product pages (e.g. `CM7-V Series`) show a **Base/SIE/SED/FIPS-SED model-number table** confirming the same base code changes one internal character to encode security variant (e.g. `KCMY1VUG12T8` base → `KCMYXVUG12T8` SIE → `KCMYDVUG12T8` SED → `KCMYFVUG12T8` FIPS SED). | The `K` + series letters (`CM`/`PM`/`CD`/`XD`/`FL`) + generation digit reliably identify product line and rough generation; the trailing form-factor code (`UG`=2.5in, `JE`=E3.S, observed consistently across CM7/CM9/CD8/CD9P datasheets) and the numeric capacity token both decode reliably. The base-vs-security-variant single-character swap (`Y`→`X`/`D`/`F`) is **officially documented per family page**, giving a clean base-model → full-SKU-variant mapping for security options specifically. | No single legend explains every internal letter position generically across all Kioxia series — the DWPD/mixed-vs-read-intensive distinction (`V` vs `R` in the model string) and form-factor letters must be cross-checked per datasheet rather than assumed universal, since letter reuse differs by generation (`Y1`/`71`/`9X` generation codes are not sequential or self-explanatory). | **Official, per-family.** Base→security-variant suffix relationship is **officially documented**; full cross-series generic grammar is **not consolidated in one official reference**. |
| **WD / SanDisk** | No official generic SSD decoder (the HDD-era *WD Model Numbers Info Sheet* referenced in the companion HDD report covers HDDs only, not SSDs). Official per-family datasheets (e.g. WD Blue 3D NAND SATA SSD data sheet) list exact model numbers per capacity/form-factor: `WDS200T2B0A`/`WDS100T2B0A`/`WDS500G2B0A`/`WDS250G2B0A` (2.5"/7mm) vs the `…B0B` suffix for M.2 2280. As of Jan 2026, WD-branded SSDs are being **rebranded to SanDisk "Optimus"** naming (Ars Technica, hwcooling.net), with vendor confirmation that **model numbers are retained unchanged** across the rebrand for continuity. | `WDS` prefix (WD SSD) plus a capacity token (`200T`/`100T`/`500G`/`250G`) plus a 4-character suffix block reliably identify brand + capacity; the trailing letter (`A` vs `B` in the WD Blue example) reliably distinguishes form factor (2.5"/7mm cased vs M.2 2280) within one family, corroborated across the WD Blue datasheet. | The generic suffix-letter meaning is **not shown to generalize across WD SSD families** the same way Kingston's or Micron's legends do — each family datasheet must be consulted individually, and there is no official document tying suffix letters to a universal grammar. The ongoing WD→SanDisk Optimus rebrand adds near-term identity-mapping churn (same MPN, different brand name on the box) that a matcher must account for. | **Official, per-family only** (mirrors the HDD-side finding that WD is the most fragmented major vendor). No generic public legend found. |
| **Seagate (Nytro)** | Seagate's official *ST Model Number Cheat Sheet* (already cited in the companion HDD report) explicitly lists `FX`/`FM` as the **enterprise-SSD segment codes** for `ST`-prefixed model numbers, but current Nytro-branded enterprise SSDs use a **separate `XA`/`XS`/`XP` prefix grammar** documented only inside individual official Nytro product manuals (Nytro 1351/1551 SATA → `XA…`; Nytro 2050/3050/3x32 SAS → `XS…`; Nytro 5060 NVMe → `XP…`), a pattern that is consistent across every Nytro product manual reviewed but never stated as an explicit generic legend in one place. | The prefix letter (`XA`/`XS`/`XP`) reliably signals interface family (SATA/SAS/NVMe respectively — inferred consistently, not a stated Seagate legend) and the numeric token that follows is the capacity in GB. Family product manuals reliably map full part numbers to capacity, SED/FIPS/ISE security variant, and interface generation. | There is **no consolidated official cross-family legend** for the `XA`/`XS`/`XP` scheme the way the ST cheat sheet exists for HDDs — the interface-family inference, while consistent across every manual reviewed, is not a documented Seagate statement and should be flagged `[unverified — pattern inferred, not vendor-stated]` if used as a hard validation rule. | **Official data (per-manual), inferred-but-uncorroborated generic legend.** The base `ST`-prefix cheat sheet's `FX`/`FM` SSD codes are official but apply to a different (older/ST-branded) SSD lineage than current Nytro parts. |
| **Kingston** | **Official, current, dedicated decoder page**: `kingston.com/en/solutions/ssd-part-number-decoder`, covering both **Client** (`SA…`/`A…`/`KC…`/`HyperX…` series letters) and **Enterprise** (`SEDC…`, "Enterprise Data Center" series) grammars, character-position by character-position, including version digit, model digit, interface code, usage-feature block, form-factor code, capacity token, and multi-pack/bundle suffix. Confirmed against real enterprise datasheets (DC600M/DC600ME `SEDC600M/480G` … `/7680G`). | This is the **most completely and currently documented vendor legend in the entire SSD survey**: series, version, interface (SATA vs NVMe), form factor, capacity, and bundle/multi-pack suffix all decode directly and officially from the string, for both client and enterprise lines. | Kingston's own page caveats: "this is a representation of **a majority** of Kingston SSDs, naming conventions **may vary**" — so treat it as high-confidence but not 100% exhaustive across every historical or acquired product line. | **Fully official**, current, and the reference implementation other vendors' decoders should be modeled after. |

### Cross-cutting pattern for the "MPN grammar decoder" plugin contract

Across every vendor above, the same two-tier structure recurs and should drive the plugin design: a **base/generic model code** (family + capacity + rough form-factor/interface, usually decodable from the string alone with an official or well-corroborated community pattern) versus a **full orderable/region/packaging/security SKU** (adds retail-vs-OEM, region, security (SED/FIPS/OPAL), or bundle suffixes that are **rarely documented in one generic legend** and instead require the specific family datasheet or PCN). Kioxia's Base/SIE/SED/FIPS-SED table and Kingston's bundle-suffix block are the two cleanest officially-documented examples of this base→variant relationship; Samsung's retail-vs-OEM suffix and Seagate's SED/FIPS/ISE letters are the least-documented (community-inferred only). The plugin contract in the prompt — *validate a token is plausibly a real MPN for that vendor, and derive family/capacity/generation from the code structure alone* — is achievable for all eight vendors at the base-code level; **exact endurance/warranty/security semantics should always defer to the datasheet**, exactly as specified, since no vendor here publishes a generic suffix legend precise enough to trust standalone.

## Recommendation

Given the OSS license-allowlist CI gate (MIT/BSD-2/BSD-3/Apache-2.0/LGPL-3.0 only, no GPL) and hw-radar's own MIT license:

1. **For SMART/health interpretation, do not vendor `drivedb.h`.** If hw-radar ever needs SMART-attribute semantics (distinct from the spec catalog itself), shell out to a system-installed `smartctl` binary at runtime and consume its (JSON) output as data. This keeps GPL-2.0-or-later content entirely outside the git repository and outside anything `uv`/PyPI-based license tooling would need to allowlist, since no GPL package is added as a dependency — only an operational assumption that the deployment host has `smartmontools` installed as a system package (already implied by the Debian 13 LXC deployment target per ADR-0003).
2. **For the product_alias / spec catalog bootstrap itself, do not treat any single external dataset as sufficient.** Build the catalog primarily from **first-party vendor datasheets/family pages** (already the pattern established for HDDs), using the per-vendor MPN grammar table above as a *validation and field-extraction* layer (does this token look like a real Samsung/Micron/Kioxia/etc. MPN, and what family/capacity/form-factor/interface can be read off the string alone), not as a spec source of truth. This matches the plugin contract in the prompt exactly: grammar-level validation and coarse decoding from the code, full variant semantics from datasheets.
3. **Open Icecat is worth a bounded trial as a runtime enrichment call** (not vendored data) for filling gaps where a manufacturer datasheet is hard to source quickly — call it live, cache the response locally (already the general caching posture recommended in the HDD report), and display Icecat's attribution per its terms. Do not commit Icecat-sourced datasheet text/specs into the repository as fixtures or seed data until the redistribution boundary in its Content License is clarified directly with Icecat (the "Client Service" framing suggests this is intended for exactly this kind of use, but that has not been confirmed for a personal/non-commercial self-hosted tool specifically).
4. **Skip TechPowerUp's SSD database as an automated source for now.** It has no self-serve API for non-academic use of the SSD dataset specifically, and unauthorized scraping would conflict with the same acquisition-tiering principle (prefer APIs/structured data, skip rather than uncontrolled-scrape) already governing hw-radar's marketplace acquisition strategy. Revisit only if a direct licensing conversation with the maintainer produces usable terms.
5. **Skip PCPartPicker and any WikiDevi-style scrape target** — no public API exists, and the operator has explicitly declined to publish one.
